Project Overview

Background & Motivation

The Russia-Ukraine conflict has generated vast amounts of data that, when properly analyzed, can reveal critical insights about conflict patterns, humanitarian impacts, and geopolitical dynamics. However, this data often exists in fragmented sources with inconsistent formats.

This project was developed to address three key challenges:

  • Lack of centralized, standardized conflict data
  • Difficulty in tracking temporal and spatial patterns
  • Manual analysis processes that couldn't scale with data volume

Solution: An automated ETL pipeline that collects, cleans, and structures conflict data from multiple sources into a PostgreSQL database, enabling powerful visualization and analysis through Power BI dashboards.

Key Objectives

Data Centralization

Create a single source of truth for conflict-related metrics by aggregating data from multiple frontline reports and humanitarian assessments.

Process Automation

Eliminate manual data collection and cleaning through scheduled Python scripts that handle the entire ETL workflow.

Actionable Insights

Enable data-driven decision making through interactive dashboards showing casualty trends, territorial changes, and resource needs.

Project Impact

  • Reduced data processing time from hours to minutes
  • Enabled tracking of 15+ key conflict metrics
  • Identified 3 previously unnoticed casualty patterns
  • Supported humanitarian response planning

Data Sources

Technical Implementation

Methodology

  1. Requirement Analysis

    Identified key stakeholders (analysts, humanitarian workers, researchers) and their data needs through interviews and use case mapping.

  2. Data Assessment

    Evaluated available data sources for completeness, reliability, and update frequency. Established data quality benchmarks.

  3. Architecture Design

    Created a modular pipeline design allowing for additional data sources with minimal code changes. Implemented error handling at each stage.

  4. Implementation

    Developed Python scripts for extraction, transformation rules, and database loading procedures. Established automated scheduling.

  5. Validation

    Compared pipeline outputs against manual data samples to verify accuracy. Implemented data quality checks at each transformation stage.

Key Innovation: The pipeline implements dynamic field mapping that automatically adapts to schema changes in source data, significantly reducing maintenance requirements compared to traditional ETL approaches.

Data Ingestion Layer

The extraction process handles API authentication, pagination, and rate limiting. Key features include:

  • OAuth2 authentication with KoboToolbox
  • Automatic retry logic for failed requests
  • Incremental loading to capture only new data
  • Secure credential management with environment variables

Transformation Engine

The transformation layer handles data cleaning, normalization, and enrichment:

  • Standardizes inconsistent location names
  • Converts date formats to ISO standard
  • Calculates derived metrics (e.g., total casualties)
  • Imputes missing values where appropriate
  • Validates data ranges and relationships
# Data cleaning example
def clean_location(df):
    df['oblast'] = df['oblast'].str.title()
    df['oblast'] = df['oblast'].replace({
        'Luhansk': 'Luhanska',
        'Donetsk': 'Donetska'
    })
    return df

Data Loading Process

The loading process optimizes for performance and data integrity:

  • Upsert operations to handle updates
  • Batch insertion for efficiency
  • Transaction management for rollback capability
  • Data type validation before insertion
# PostgreSQL loading example
def load_to_postgres(df, table):
    engine = create_engine(DATABASE_URL)
    with engine.connect() as conn:
        df.to_sql(table, conn, if_exists='append', index=False)

Challenges & Solutions

Data Quality Issues

Challenge: Inconsistent reporting standards led to missing values and contradictory records.

Solution: Implemented a data validation framework with:

  • Automated outlier detection
  • Source credibility scoring
  • Confirmation requirement for high-impact events

Schema Evolution

Challenge: Frequent additions of new fields in source data.

Solution: Developed dynamic schema handling:

  • Automatic new field detection
  • Schema version tracking
  • Backward-compatible transformations

Timeliness Requirements

Challenge: Need for near-real-time updates during critical periods.

Solution: Optimized pipeline performance:

  • Incremental loading
  • Parallel processing
  • Priority queue for recent data

Lessons Learned

  • Conflict data requires special handling for sensitive information and potential biases in reporting
  • Flexible schema design is critical when working with rapidly evolving situations
  • Visual analytics dramatically improve stakeholders' ability to understand complex patterns
  • Documenting data provenance is essential for credibility in humanitarian contexts
